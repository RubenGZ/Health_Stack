Configuración de Oracle Cloud Free Tier

0.1 Crear cuenta

Accede a https://signup.cloud.oracle.com y completa el registro. Solo necesitas un correo válido, teléfono para SMS y tarjeta de crédito/débito para verificación (no se cobra nada).

⚠️

Oracle realiza un cargo de verificación de $1 USD que se devuelve automáticamente en 24-48h. Nunca se te cobrará mientras uses solo recursos Always Free.

Registro inicial

Completa nombre, país, correo y tipo de cuenta (Personal). Elige la región más cercana a ti — esta decisión es permanente.

Verificación de identidad

Confirma el correo, verifica el teléfono por SMS e introduce los datos de la tarjeta. El proceso tarda ~5 minutos.

Activar 2FA (recomendado)

Ve a Profile → Security Settings → Enable 2-Factor Authentication para proteger tu cuenta.

0.2 Crear la instancia VM

Navega a Compute → Instances → Create Instance y usa esta configuración exacta:

¿Por qué A1.Flex? Oracle regala hasta 4 OCPU ARM + 24 GB RAM totales. Con 2 OCPU + 12 GB tienes margen para crear una segunda instancia gratuita en el futuro.

# Mostrar la clave pública para pegarla en la consola de Oracle

0.4 Configurar Security Lists (Firewall Oracle)

Navega a Networking → VCN → Security Lists → Default → Add Ingress Rules:

🚫

NO abrir el puerto 8000 en Security Lists. FastAPI escucha solo en 127.0.0.1:8000 y se expone exclusivamente a través de Nginx.

Preparación del Entorno de Producción

2.1 Configurar PostgreSQL

bash + sql
sudo -u postgres psql << 'SQL'
CREATE USER health_user WITH PASSWORD 'SuperSecretPassword123!';
CREATE DATABASE health_stack_db OWNER health_user;
GRANT ALL PRIVILEGES ON DATABASE health_stack_db TO health_user;
\q
SQL

# Verificar conexión
psql -U health_user -d health_stack_db -h 127.0.0.1

2.2 Archivo .env de producción

bash — crear /etc/health-stack/.env
sudo mkdir -p /etc/health-stack/keys
sudo chown ubuntu:ubuntu /etc/health-stack -R

cat > /etc/health-stack/.env << 'EOF'
# ── Aplicación ──────────────────────────────────────
APP_ENV=production
APP_NAME=Health Stack
DEBUG=false
ALLOWED_HOSTS=["tu-dominio.com","www.tu-dominio.com"]

# ── Base de datos ─────────────────────────────────────
DATABASE_URL=postgresql://health_user:SuperSecretPassword123!@127.0.0.1:5432/health_stack_db
DB_POOL_SIZE=10
DB_MAX_OVERFLOW=20

# ── JWT RS256 ─────────────────────────────────────────
JWT_ALGORITHM=RS256
JWT_PRIVATE_KEY_PATH=/etc/health-stack/keys/private.pem
JWT_PUBLIC_KEY_PATH=/etc/health-stack/keys/public.pem
JWT_ACCESS_TOKEN_EXPIRE_MINUTES=30
JWT_REFRESH_TOKEN_EXPIRE_DAYS=7

# ── Seguridad ─────────────────────────────────────────
SECRET_KEY=cambia-esto-por-una-cadena-aleatoria-muy-larga
CORS_ORIGINS=["https://tu-dominio.com"]

# ── Logs ──────────────────────────────────────────────
LOG_LEVEL=INFO
LOG_FILE=/var/log/health-stack/app.log
EOF

2.3 Generar claves RSA para JWT RS256

bash — claves RSA
# Generar clave privada RSA 4096 bits
openssl genrsa -out /etc/health-stack/keys/private.pem 4096

# Extraer clave pública
openssl rsa -in /etc/health-stack/keys/private.pem \
            -pubout \
            -out /etc/health-stack/keys/public.pem

# Asegurar permisos
chmod 600 /etc/health-stack/keys/private.pem
chmod 644 /etc/health-stack/keys/public.pem

# Verificar
ls -la /etc/health-stack/keys/

Deploy Paso a Paso en Oracle Cloud

4.1 Clonar el repositorio

bash
cd /opt
sudo mkdir health-stack && sudo chown ubuntu:ubuntu health-stack
git clone https://github.com/TU_USUARIO/health-stack.git /opt/health-stack

4.2 Entorno virtual y dependencias

bash
cd /opt/health-stack
python3.11 -m venv venv
source venv/bin/activate
pip install --upgrade pip
pip install -r requirements.txt

# Crear directorio de logs y enlazar .env
sudo mkdir -p /var/log/health-stack
sudo chown ubuntu:ubuntu /var/log/health-stack
ln -s /etc/health-stack/.env /opt/health-stack/.env

4.3 Migraciones Alembic

bash
cd /opt/health-stack
source venv/bin/activate

# Ver estado actual
alembic current

# Aplicar todas las migraciones
alembic upgrade head

# Verificar tablas creadas
psql -U health_user -d health_stack_db -h 127.0.0.1 -c "\dt"

4.4 Servicio systemd para FastAPI

systemd service
sudo tee /etc/systemd/system/health-stack.service > /dev/null << 'EOF'
[Unit]
Description=Health Stack FastAPI Application
After=network.target postgresql.service
Requires=postgresql.service

[Service]
Type=exec
User=ubuntu
WorkingDirectory=/opt/health-stack
Environment="PATH=/opt/health-stack/venv/bin"
EnvironmentFile=/etc/health-stack/.env
ExecStart=/opt/health-stack/venv/bin/uvicorn app.main:app \
    --host 127.0.0.1 \
    --port 8000 \
    --workers 4 \
    --log-level info
Restart=on-failure
RestartSec=5
NoNewPrivileges=yes
PrivateTmp=yes

[Install]
WantedBy=multi-user.target
EOF

sudo systemctl daemon-reload
sudo systemctl enable health-stack
sudo systemctl start health-stack
sudo systemctl status health-stack

4.5 Configurar Nginx como reverse proxy

nginx
sudo tee /etc/nginx/sites-available/health-stack > /dev/null << 'EOF'
# HTTP → HTTPS redirect
server {
    listen 80;
    server_name tu-dominio.com www.tu-dominio.com;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl http2;
    server_name tu-dominio.com www.tu-dominio.com;

    ssl_certificate     /etc/letsencrypt/live/tu-dominio.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/tu-dominio.com/privkey.pem;
    include             /etc/letsencrypt/options-ssl-nginx.conf;

    # Security headers
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
    add_header X-Frame-Options DENY always;
    add_header X-Content-Type-Options nosniff always;

    root /opt/health-stack/frontend;
    index index.html;

    # Frontend SPA
    location / {
        try_files $uri $uri/ /index.html;
        expires 1h;
    }

    # Static assets con caché larga
    location ~* \.(js|css|png|jpg|ico|svg|woff2)$ {
        expires 30d;
        add_header Cache-Control "public, immutable";
    }

    # API Backend → FastAPI
    location /api/ {
        proxy_pass         http://127.0.0.1:8000;
        proxy_http_version 1.1;
        proxy_set_header   Host $host;
        proxy_set_header   X-Real-IP $remote_addr;
        proxy_set_header   X-Forwarded-Proto $scheme;
        proxy_read_timeout 60s;
        client_max_body_size 10M;
    }
}
EOF

sudo ln -s /etc/nginx/sites-available/health-stack /etc/nginx/sites-enabled/
sudo rm -f /etc/nginx/sites-enabled/default
sudo nginx -t && sudo systemctl reload nginx

4.6 Certificado SSL con Certbot

Opción A — Dominio propio

bash
sudo certbot --nginx -d tu-dominio.com -d www.tu-dominio.com \
    --email tu@email.com --agree-tos --no-eff-email

# Verificar renovación automática
sudo certbot renew --dry-run

Opción B — DuckDNS (dominio gratuito)

bash
# 1. Crear cuenta en https://www.duckdns.org
# 2. Crear subdominio: healthstack.duckdns.org → apuntar a tu IP

pip install certbot-dns-duckdns
sudo tee /etc/letsencrypt/duckdns.ini << EOF
dns_duckdns_token=TU_TOKEN_AQUI
EOF
sudo chmod 600 /etc/letsencrypt/duckdns.ini

sudo certbot certonly \
  --authenticator dns-duckdns \
  --dns-duckdns-credentials /etc/letsencrypt/duckdns.ini \
  -d healthstack.duckdns.org \
  --email tu@email.com --agree-tos

4.8 Backups automáticos de PostgreSQL

bash — backup script + cron
sudo tee /usr/local/bin/backup-health-db.sh > /dev/null << 'EOF'
#!/bin/bash
BACKUP_DIR="/var/backups/health-stack"
DATE=$(date +%Y%m%d_%H%M%S)
mkdir -p "$BACKUP_DIR"

PGPASSWORD="SuperSecretPassword123!" pg_dump \
    -U health_user -h 127.0.0.1 -d health_stack_db \
    --format=custom --compress=9 \
    -f "$BACKUP_DIR/health_stack_db_${DATE}.dump"

# Mantener solo los últimos 7 días
find "$BACKUP_DIR" -name "*.dump" -mtime +7 -delete
echo "[$(date)] Backup: health_stack_db_${DATE}.dump"
EOF

sudo chmod +x /usr/local/bin/backup-health-db.sh

# Cron: backup diario a las 3:00 AM
(crontab -l 2>/dev/null; echo "0 3 * * * /usr/local/bin/backup-health-db.sh >> /var/log/health-stack/backup.log 2>&1") | crontab -

Seguridad y Buenas Prácticas

5.1 Configurar Fail2ban

bash
sudo tee /etc/fail2ban/jail.local << 'EOF'
[DEFAULT]
bantime  = 3600
findtime = 600
maxretry = 5

[sshd]
enabled  = true
maxretry = 3
bantime  = 86400

[nginx-http-auth]
enabled  = true
logpath  = /var/log/nginx/health-stack-error.log
EOF

sudo systemctl enable fail2ban
sudo systemctl restart fail2ban
sudo fail2ban-client status

5.2 Rate limiting en Nginx

nginx
# En /etc/nginx/nginx.conf, bloque http {}
limit_req_zone $binary_remote_addr zone=api_limit:10m rate=30r/m;
limit_req_zone $binary_remote_addr zone=login_limit:10m rate=5r/m;

# En location /api/:
limit_req zone=api_limit burst=10 nodelay;

# Para endpoint login:
location /api/v1/auth/login {
    limit_req zone=login_limit burst=3 nodelay;
    proxy_pass http://127.0.0.1:8000;
}

5.3 Actualizaciones automáticas de seguridad

bash
sudo tee /etc/apt/apt.conf.d/20auto-upgrades << 'EOF'
APT::Periodic::Update-Package-Lists "1";

EOF

🖥️ Servidor
  • Ubuntu 22.04 actualizado (apt update && upgrade)

Troubleshooting

Error   Conexión a la base de datos falla ▼
bash — diagnóstico
# Verificar que PostgreSQL escucha en el puerto
sudo ss -tlnp | grep 5432

# Probar conexión directa
psql -U health_user -d health_stack_db -h 127.0.0.1

# Ver logs de PostgreSQL
sudo tail -f /var/log/postgresql/postgresql-14-main.log

# Comprobar DATABASE_URL en el .env
grep DATABASE_URL /etc/health-stack/.env
Error   Nginx 403/404 en archivos estáticos ▼
bash
# Arreglar permisos del directorio frontend
sudo chmod -R 755 /opt/health-stack/frontend/
sudo chown -R ubuntu:www-data /opt/health-stack/frontend/

# Verificar que nginx puede leer los archivos
sudo -u www-data ls /opt/health-stack/frontend/
Error   El servicio systemd no arranca ▼
bash
# Ver últimos logs de error
sudo journalctl -u health-stack --no-pager -n 50

# Arrancar manualmente para ver el error completo
cd /opt/health-stack && source venv/bin/activate
uvicorn app.main:app --host 127.0.0.1 --port 8000

# Causas comunes:
# → ModuleNotFoundError: pip install -r requirements.txt
# → .env no encontrado: verificar enlace simbólico
# → Puerto 8000 en uso: sudo lsof -i :8000
Error   Certbot no puede obtener certificado SSL ▼
bash
# Verificar que el dominio apunta a la IP correcta
dig +short tu-dominio.com  # Debe devolver la IP de Oracle

# Verificar que Nginx está activo y el puerto 80 abierto
sudo systemctl status nginx
sudo ufw status | grep 80

# Alternativa: validación DNS en vez de HTTP
sudo certbot certonly --manual \
  --preferred-challenges dns \
  -d tu-dominio.com
Error   Respuesta lenta o errores 502 Bad Gateway ▼
bash
# Verificar que FastAPI responde internamente
curl http://127.0.0.1:8000/api/v1/health

# Monitorizar recursos
htop  # Si CPU/RAM al límite → aumentar workers o escalar VM

# Logs de FastAPI para errores internos
sudo journalctl -u health-stack --since "10 minutes ago"

# Aumentar timeout en nginx si hay operaciones lentas:
# proxy_read_timeout 120s;
